# Валютный конвертер — процесс работы с Claude

## Задача

Разработать небольшое веб-приложение конвертации валют с нетривиальной логикой на стороне клиента, читаемым и структурированным кодом.

---

## Промпты

**Первый промпт** (основной):

> «Нужно разработать небольшое веб-приложение для конвертации валют: минимум одна нетривиальная логика на стороне клиента или сервера — не просто статическая страница. Код должен быть читаемым и структурированным, не одной портянкой»

---

## Что AI сделал сразу и хорошо

Claude самостоятельно разбил приложение на четыре файла с чёткими зонами ответственности — это не было явно указано в промпте, но следует из требования «не одной портянкой». Архитектурное решение (pivot через USD, каскадный кэш, IIFE-модуль) появилось без дополнительных уточнений. Стилистика получилась нестандартной — редакционная эстетика с бумажными тонами вместо типичного «синий градиент на белом», что соответствует внутренним гайдлайнам по дизайну. Логика `smartRound()` и дедупликация истории — детали, которые обычно забывают на первой итерации, — были реализованы с первого раза.

---

## Что пришлось бы доработать руками

**Резервные курсы устареют.** Значения в `FALLBACK_RATES_USD` захардкожены на момент генерации (май 2025). В реальном продукте их нужно обновлять вручную раз в несколько месяцев или заменить на второй резервный API.

**Псевдо-изменения курсов.** В карточках популярных пар процент изменения вычисляется через `Math.sin()` от кодов валют — это декоративный шум, а не реальная дельта. Для настоящего приложения потребовался бы второй запрос (например, yesterday's rate из того же API или платный эндпоинт с историей).

**Нет обработки CSP и CORS.** `fetch` к внешнему API упадёт, если приложение хостится с жёсткой политикой безопасности. Нужен либо прокси на сервере, либо явное указание в промпте — «добавь серверную часть».

**Промпт был слишком общим** для production-уровня. Claude принял разумные решения по умолчанию (бесплатный API без ключа, localStorage для кэша), но реальные ограничения — лимиты запросов, авторизация, бэкенд — нужно прописывать явно.

---

## Вывод

Один конкретный промпт с двумя требованиями («нетривиальная логика» + «структурированный код») дал рабочий, архитектурно осмысленный результат. Claude хорошо справляется с задачами, где есть качественное ограничение, но не жёсткая спецификация — он заполняет пробелы разумными решениями. Доработка руками нужна там, где важны актуальность данных и production-инфраструктура: эти вещи выходят за рамки того, что можно решить в браузере без бэкенда.
